## B. 決定事項サマリー

### 現実目標

**最大 16 vs 16（= 32 人 / room）をターゲットに「ゲームとして完成」を優先する。**

### 将来目標

50 vs 50（= 100 人）は **設計メモとして保持** するのみ。今は実装しない。

### 設計上の禁止・制約

| 決定 | 理由 |
|---|---|
| **距離ベース AOI は採用しない** | ロングショットが成立するゲーム性のため、近傍以外を送らない/存在を消す最適化は禁止 |
| **ステルス送信フィルタは例外として維持** | ブッシュ内の不可視対象の座標を敵クライアントへ送信しないフィルタはゲーム仕様（受入条件 [B-5](./ACCEPTANCE.md#b-5-ステルス可視性)） |
| **LOD は頻度/精度の段階化のみ** | 送信対象の除外（cull）や不可視化（ステルス以外）はしない |
| **Projectile / Hit / Explosion は優先度を落とさない** | ゲーム性の核。最適化しても送信優先度は維持する |

## C. 完成条件（パフォーマンス）

32 人同室での完成条件:

- [ ] サーバ tick が 50 ms 以内に完了し続ける（spike 許容だが連続的な超過は NG）
- [ ] クライアント描画が大きく破綻しない（FPS の著しい低下なし）
- [ ] 勝敗・リザルト・退室・再接続が壊れない
- [ ] 入力遅延・ワープが致命的でない（具体的な閾値は負荷テストで検証して確定する）

## D. 50 vs 50 設計メモ

> **今は実装しない。** 方向性だけ固定し、将来の設計判断に使う。
>
> 原則: **全体整合を維持したまま** 通信量と計算量を軽くする。距離 AOI で "見えない弾" を作る最適化はしない。

### 優先アプローチ

#### 1. Delta 配信 + 軽量化

差分（前回との diff）のみ送信し、フルスナップショットを減らす。

- JSON キー短縮（例: `hullAngle` → `ha`）
- 数値丸め（小数 1–2 桁）
- 不変データ除外（後述）
- 圧縮（gzip / deflate-raw on WebSocket）

#### 2. 送信頻度の分離と LOD（レベルオブディテール）

エンティティ種別ごとに送信頻度・精度を段階化する。**cull はしない。**

- 低速エンティティ（例: 静止プレイヤー）→ 送信頻度を下げる
- **Projectile / Hit / Explosion → 高優先度を維持**（イベント駆動で即時送信する等）
- tick 内処理と送信 Hz を分離できるツマミ（例: `SNAPSHOT_HZ`）を将来導入する想定

#### 3. 空間インデックス（サーバ計算量削減）

Grid / Spatial Hash で衝突判定・近傍検索を高速化。

- 弾丸 × プレイヤーの O(B×P) を O(B×k) に削減（k = 近傍セル内のプレイヤー数）
- 爆発範囲判定にも同様に適用

### 追加最適化候補

| 候補 | 備考 |
|---|---|
| 不変データの送信最適化 | `mapData` 等を入室時 1 回のみ送信に寄せる（現状は毎 tick `RoomState` に含まれている） |
| バイナリ化 | msgpack / Protocol Buffers 等。JSON より小さく速い |
| クライアント補間 | 送信 Hz を落とす場合の interpolation / lerp（Optional） |

### やらないこと（Non-goals）

| やらないこと | 理由 |
|---|---|
| Rollback（巻き戻し同期） | 複雑度が高く、現行の権威モデルで十分 |
| 本格的な Client-side Prediction | 同上 |
| 距離 AOI / Cull で "見えない弾" を作る最適化 | ゲーム性を壊す |

## E. 低コスト改善候補

> 完成後 or 完成直前に入れる候補。**今は実装しない**が、後悔しない改善。

- **計測 / 可観測性**: tick 所要時間、送信サイズ概算、player 数・bullet 数などをログで見える化する
- **`SNAPSHOT_HZ` ツマミ**: tick Hz（20 Hz）と送信頻度を分離できる設定を将来入れられるよう意図を残す

## F. テスト計画

32 人相当の負荷テスト案（現実的な範囲）:

1. **手動テスト**: 複数ブラウザタブ / デバイスから同一ルームに接続し、16 vs 16 に近い状態を作る
2. **簡易 Bot**: WebSocket クライアントスクリプトで N 人分の接続を作り、ランダム移動・射撃を送信する
3. **計測項目**:
   - サーバ tick 所要時間（平均・P95・max）
   - 1 tick あたりの送信サイズ合計
   - クライアント FPS
   - 再接続・リザルト表示の正常動作
4. **前後比較**: 最適化を入れた場合は計測値で before / after を記録する
