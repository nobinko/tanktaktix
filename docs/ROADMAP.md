## Phase 3.5: 仕様修正（コード変更必須） - **完了**

### B-5修正: 射撃による隠密解除を廃止する (済)

> **仕様変更の意図**: 「ブッシュ内で射撃すると1000ms間だけ敵から可視になる」という現行動作を廃止する。新仕様では**ブッシュ内にいる間は常に隠密状態**であり、射撃しても解除されない。隠密の解除条件は「ブッシュ外に出ること」のみとする。

#### 現行コードの動作（変更前）

`server/src/index.ts` の tick 処理内で以下のロジックが動いている:

```typescript
// L159: 定数
const REVEAL_DURATION_MS = 1000;

// L1015: tryShoot() 内
p.lastFiredAt = now;

// L1363-1366: 毎 tick の隠密更新
const inBush = isPointInBush(p.x, p.y, room.mapData.walls);
const revealByShooting = (now - p.lastFiredAt < REVEAL_DURATION_MS);
p.isHidden = inBush && !revealByShooting;
```

また `shared/src/index.ts` の `PlayerSummary` 型に `lastFiredAt: number` フィールドが存在する。

#### 実装指示（AIへ）

以下を**正確に**変更すること。それ以外は触れない。

**`server/src/index.ts`**

1. `const REVEAL_DURATION_MS = 1000;` の行を**削除**する
2. `PlayerRuntime` 型定義の `lastFiredAt: number;` フィールドを**削除**する
3. `tryShoot()` 関数内の `p.lastFiredAt = now;` を**削除**する
4. 隠密更新ロジックを以下に置き換える:

```typescript
// 変更前
const inBush = isPointInBush(p.x, p.y, room.mapData.walls);
const revealByShooting = (now - p.lastFiredAt < REVEAL_DURATION_MS);
p.isHidden = inBush && !revealByShooting;

// 変更後
const inBush = isPointInBush(p.x, p.y, room.mapData.walls);
p.isHidden = inBush;
```

**`shared/src/index.ts`**

5. `PlayerSummary` 型の `lastFiredAt: number;` フィールドを**削除**する

#### 変更不要なもの

- `client/src/main.ts` の描画ロジックは `isHidden` boolean のみを参照しており変更不要
- HUD の「🕵️ HIDDEN」表示も変更不要

#### 完了確認

変更後に以下を実行してゼロ件であることを確認すること:

```bash

grep -r "REVEAL_DURATION_MS" server/
```

## Phase 4: アイテムシステム刷新・新アイテム・フラッグ改善

### 4-1. アイテム総量管理（マップ別固定プール） - **[x] 完了**

> **設計意図**: 旧TM系ゲームの仕様に準拠。無限スポーンだとアイテムが溢れてゲームバランスが崩れる。マップごとに上限を固定することで「どこにアイテムがあるか」が読める戦略性を生む。

現行のランダム無限スポーンをやめ、**マップごとにアイテム総量を固定する**。

- 各マップに「アイテムプール」を定義する
- ゲーム開始時にプール分のアイテムをマップ上にランダム配置する
- アイテムが**取得またはショットで破壊**されると、同種のアイテムが別のランダム地点に再スポーンする
- マップ上に同時存在できる数 = プールで定義した総量（上限を超えて増えない）

**暫定プール（全マップ共通・マップ設計時に調整予定）:**

| アイテム | 数量 |
|---|---|
| medic | 2 |
| ammo | 2 |
| heart | 2 |
| bomb | 2 |
| rope | 2 |
| boots | 2 |
| **合計** | **12** |

### 4-2. アイテムへの当たり判定 - **[x] 完了**

> **設計意図**: 弾をアイテムに当てて敵に使わせない、有利な位置に再スポーンさせるなどの戦略的プレイを可能にしたい。アイテムが「守るべきもの」になることでマップ上の動きが豊かになる。

- 弾丸がアイテムに命中するとアイテムが**破壊**される
- 破壊されたアイテムは 4-1 のルールに従い別地点に再スポーンする
- フラッグにも同様の当たり判定を追加する（後述）

### 4-3. 所持上限チェック - **[x] 完了**

> **設計意図**: 現行は満タン状態でもアイテムを踏めてしまい意図しない取得が起きている。上限を設けることで「今取るべきか」「取れないから後回し」という判断が生まれ、アイテム争奪の緊張感を作る。

所持上限を超える場合はアイテムを取得できない。

| アイテム | 所持上限 |
|---|---|
| medic / heart | HP が 100% の場合は取得不可 |
| ammo | 40発（初期20発、上限40発） |
| rope | 2本（ダブルロープ） |
| bomb | 1個 |
| boots | 1個 |
| flag | 1本（CTF） |

### 4-4. 新アイテム定義 - **[x] 完了**

> **設計意図**: 既存の medic（部分回復）・ammo（弾薬補充）に対して戦略の幅を広げる4種を追加する。heart は大ピンチ時の逆転札、bomb はマップコントロール用の高リスク兵器、rope はフラッグ戦を劇的に変えるリーチ拡張アイテム、boots はオリジナルの機動力バフ（旧TM系には存在しない）。それぞれ明確に異なる用途を持たせることでアイテム争奪の動機を分散させる。

#### heart（ハート）
- 効果: HP を **100% まで全回復**
- 上限: HP が 100% の場合は取得不可

#### bomb（ボム）
- 効果: **次の1発**がボムショットになる
- 弾速は通常弾と同じ
- 爆風半径は通常の **3倍**（現行 40px → 120px）
- ダメージゾーン（爆心地からの距離で3段階）:
  - 内側 1/3 以内（0〜40px）: **60% ダメージ**
  - 中間 1/3（40〜80px）: **40% ダメージ**
  - 外側 1/3（80〜120px）: **20% ダメージ**
- 所持上限: 1個（ボムショット発射後にリセット）

#### rope（ロープ）
- 効果: AIMドラッグ中に `R` キーでロープを射出できる
- ロープ長: **シングル（1本所持）= 200px / ダブル（2本所持）= 300px**
- ロープでアイテム・敵フラッグ・**味方フラッグも**取得可能
- 所持上限: 2本

#### boots（ブーツ）※オリジナルアイテム
- 効果: 移動速度が **1.5倍** になる
- 持続: **3回移動到着**まで（3回消費後に効果終了）
- 所持上限: 1個

**実装詳細（AIへ）:**
- **速度倍率**: `MOVE_SPEED`（通常 6 px/tick）を boots 効果中は **9 px/tick** に変更する（`MOVE_SPEED * 1.5`）
  - boots 効果中のプレイヤーのみ速度を上げる。グローバル定数は変えない
  - 実装例: `const effectiveSpeed = p.bootsCharges > 0 ? MOVE_SPEED * 1.5 : MOVE_SPEED;`
- **カウント方法**: 「3回移動」は**目的地に到着した回数**でカウントする（moveQueue からの消化1件ごとに +1）
  - 移動中断（壁衝突など）は到着扱いにしない
  - 到着カウントが 3 に達した時点で `bootsCharges = 0` にし、速度を元に戻す

### 4-5. フラッグ挙動改善 - **[x] 完了**

> **設計意図**: 現行はドロップ直後に同じプレイヤーが即座に再取得できてしまいドロップの意味が薄い。「動くと再取得、止まると取得しない」というトレードオフを設けることで、被弾後の行動選択に緊張感を持たせる。フラッグへの弾丸当たり判定は敵フラッグを牽制する新戦術を生む。

#### 現状の問題
フラッグ保持中に被弾すると、旗がフラッグの**初期位置（`flagPositions` で定義されたベース位置）に即時リセット**される。スポーン地点と初期位置は必ずしも一致しない（別の場所にフラッグを配置するマップも想定）。ドロップという中間状態がなく、戦術的なやり取りが生まれない。

#### 改善後の挙動
1. 被弾によるフラッグドロップ → 旗がその場に落下
2. ドロップ直後: 当該プレイヤーはフラッグと位置が重なっているが**立ち止まったままでは取得しない**
3. そのプレイヤーが**移動コマンドを入力**すると、移動開始時の当たり判定でフラッグに接触 → **取得が発生し移動がキャンセル**される
4. つまり「ドロップ直後にすぐ動くと即再取得になる」というトレードオフが生まれる

#### フラッグへの当たり判定（4-2 と連動）
- 弾丸がフラッグに命中すると旗がベース位置にリセットされる（キャリー中の旗は除く）

### 4-6. ダメージ表現（Canvas描画でHP段階を視覚化） - **[ ] 未実装**

> **設計意図**: 現行は HUD の数字でしか HP を確認できず、遠くの敵の状態が直感的に把握しにくい。車体の見た目を HP に応じて変えることで「あの戦車はボロボロだから今が攻め時」という視覚的な状況判断を可能にする。砲台は HP によらず固定し、戦術的な向き判断を妨げない。

HP に応じて**Canvas描画で車体の見た目を変える**。PNGスプライトは使わない。

- 描画対象: **車体のみ**（砲塔の形状・位置は HP によらず固定）
- 実装方法: 既存の Canvas 図形描画を HP 段階ごとに分岐させる（PNG ファイル不要）
- 後日 PNG 差し替えしやすいよう「HP 段階を返すヘルパー関数」を切り出す設計にする

#### HP 段階と描画ルール

| HP | 状態 | 車体描画の変化 |
|---|---|---|
| 81〜100% | 正常 | 現行と同じ（チームカラー単色） |
| 61〜80% | 軽微な損傷 | 車体を少し暗くする（brightness 80%） |
| 41〜60% | 中程度の損傷 | 車体を暗くし、小さい黒い傷を数本描く |
| 21〜40% | 重大な損傷 | 車体を濃くし、煙エフェクト（小さなグレー円）を毎フレーム描く |
| 1〜20% | 瀕死 | 車体を最も暗くし、炎エフェクト（オレンジ小円）を毎フレーム描く |

- チームカラー（赤/青）は維持する（描画の基底色を赤/青にした上で上記を重ねる）
- **実装メモ**: PNGファイル・外部アセット不要。`client/src/main.ts` の戦車描画関数内のみ変更

### 4-7. AIMアクション派生 - **[x] 完了**

> **設計意図**: 射撃アクションを「攻撃」以外に使う選択肢を作る。アモ投げは自分の弾を削るコスト付き支援、ヘルス投げは自分のHPを削るリスク付き支援で、いずれも敵に当たっても効果が発動するため「支援か妨害か」の判断が必要になる。フラッグ投げはCTFにリレー戦術を導入する。ロープは射程外のアイテム・フラッグを取る戦術行動。いずれも既存のAIMドラッグ操作を拡張する形なので学習コストが低い。

AIMドラッグ中に特定キーを押すことで、投擲・射出などの派生アクションを実行できる。

#### AIM+R（ロープ射出）

- 前提条件: **rope を1本以上所持**していること
- 効果: AIMの向きにロープを射出し、先端が触れたアイテム・フラッグを手元に引き寄せる
- ロープ長: シングル（1本所持）= 200px / ダブル（2本所持）= 300px
- 取得可能対象: アイテム・敵フラッグ・**味方フラッグも**対象
- 使用後: rope の所持数が1本消費される（ダブル → シングル、シングル → 0本）
- クールダウン: 通常射撃と同じクールダウンが発生する

#### AIM+A（アモ投げ）

- 効果: 投擲物が命中したプレイヤー（味方・敵問わず）の弾薬 **+5**
- 消費: **自分の弾薬を 5 消費**して投げる
- 専用スプライト: アモボックスが飛ぶ画像
- 上限超過時: 受け取り側が弾薬上限（40発）を超える場合は上限でキャップ

#### AIM+H（ヘルス投げ）

- 効果: 投擲物が命中したプレイヤー（味方・敵問わず）の HP **+20%**
- 消費: **自分の HP を 20% 消費**して投げる（自分のHPから削り出す）
- 使用条件: 自分の HP が **20% 超**でないと使用不可（HP 20% 以下では投げられない）
- 専用スプライト: 回復キットが飛ぶ画像
- 上限超過時: 受け取り側が HP 100% の場合は効果なし

#### AIM+F（フラッグ投げ）

- 前提条件: CTF で**自分がフラッグを保持中**であること
- 効果: 保持中のフラッグをAIM方向に投げる
- 消費: **弾薬消費なし**
- クールダウン: 通常射撃と同じクールダウンが発生する

**着弾後の挙動（取得判定）:**
- 投擲物がフラッグを**持っていない味方**に命中 → その味方が即座にフラッグを取得（キャリア状態になる）
- 投擲物がフラッグを**持っている味方・または敵**に命中 → 着地点にフラッグが落下（通常ドロップ状態）
- 誰にも命中しなかった → 着地点にフラッグが落下（通常ドロップ状態）
- ドロップ状態のフラッグは通常の接触取得ルールに従う

### 5-7. 簡易Bot / 戦術コマンド設計
> **設計意図**: 将来の簡易Bot構想と、Botに対して「どんな思考モードを持たせたいか」を整理する拡張の設計メモ。

- [ ] **簡易Botの実装構想**: まずシンプルなbotを導入し、将来的には簡単な戦術プリセットの指示を行えるようにする。
- [ ] **想定する戦術コマンドの分類**:
  - **旗奪取（自力帰還）**: 敵旗を積極的に狙い、奪ったら自分で持ち帰って得点を狙う行動。
  - **旗奪取（パス優先）**: 敵旗を狙い、奪取後は味方へのパスを優先する（「取って渡す」中継役）。
  - **キル優先攻撃**: 旗をあえて主目的にせず、有利ポジションを取り敵を倒して圧をかける。
  - **旗防衛**: 自陣の旗を守ることを優先し、敵の旗への接近や奪取を妨害する。
  - **生存優先防衛 / 失点拒否**: 無理に前に出ず「倒されにくく」立ち回り、相手にポイントを与えないことを重視する。
