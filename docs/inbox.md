## マップ設計のノウハウ

- **対称性が基本**: 点対称 or 線対称。チーム間の公平性を担保する。ただし特殊マップもあるので「基本は」という認識が重要
- **旗は基本複数本ずつ**: 赤青各チーム複数本（2〜3本が目安）。戦略の幅が広がる（Aの旗とBの旗で攻撃分散や集中の駆け引きが生まれる）。ただし多すぎると取りやすい旗に戦力集中して面白くない。**最大3本ずつがセオリー**
- **オブジェクト配置のセンス**: 橋のそばにブッシュを置くなら、ブッシュの上下位置を橋と揃える。見た目の整合性が大事
- **スポーン地点≠旗位置**: リスポーン時に旗に埋まって動けなくなるのを防ぐ

## マップシステム刷新 — プレハブ方式

マップ1の前提となる設計。将来のマップエディタ / DB保存（5-5）も見据える。

### 設計方針

**既存の Wall[] 直指定とプレハブ配置の併用**
- MapData 定義時: `objects[]`（プレハブ）+ `walls[]`（自由配置）+ `bushes[]`（動的ブッシュ）
- ゲーム開始時に1回だけ `Wall[]` にフラット展開（動的ブッシュは別管理）
- 展開後は既存のサーバー/クライアントコードがそのまま動く
- クライアントにはプレハブ情報も送信（type別の描画・テクスチャ対応のため）
- 既存マップ（alpha〜epsilon, test S/M/L）はそのまま残す（walls[] のみで互換動作）

### プレハブ一覧（20種）

**House（家）** — 中身詰まった長方形ブロック（3種）
- `house-s`: 60 x 40（タンク1台分の遮蔽）
- `house-m`: 120 x 70（3〜4台分の遮蔽）
- `house-l`: 180 x 100（陣地レベル）

**Base（基地）** — 壁で囲まれた空間、開口部あり（4種）
- 外寸 200 x 200、壁厚 20、開口幅 60（タンク1.5台分）
- `base-1open` — 開口1つ（配置時の回転で方向指定）
- `base-2open-opposite` — 開口2つ（対面: 北南型）
- `base-2open-adjacent` — 開口2つ（隣接: 北東型）
- `base-3open` — 開口3つ

**River（川）** — 通行不可の帯、幅80（9種）
- 直線: `river-s`(200) / `river-m`(400) / `river-l`(600)
- エルボ（45度曲がり、セグメント分割でなめらか描画）:
  - `river-elbow-gentle-s` / `river-elbow-gentle-l`（緩曲率、5〜8セグメント）
  - `river-elbow-mid-s` / `river-elbow-mid-l`（中曲率、3〜6セグメント）
  - `river-elbow-sharp-s` / `river-elbow-sharp-l`（急曲率、2〜4セグメント）

**Bridge（橋）** — 川の上に配置して通行可能にする構造物（2種）
- `bridge-s`: 100 x 150（短い橋）
- `bridge-l`: 100 x 250（長い橋）
- 幅100でリバー幅80を両側10ずつ余裕持って覆う

**Oneway（一方通行壁）** — 1種
- `oneway`
- 通過可能方向は回転角度で決定（direction フィールド廃止）
- くの字配置は2枚のonewayを角度つけて並べて再現

**Bush（動的ブッシュ）** — ロープで移動可能な茂み（1種）
- `bush`（円形、半径 40）

### 静的ブッシュ（Wall type: "bush"）

- 既存の Wall type: "bush" をそのまま使用
- 自由サイズの範囲指定、移動不可
- 広い隠密ゾーンとして機能
- TM系列にない新要素

### 回転

- 全プレハブ自由角度（0〜360度）対応
- 当たり判定を AABB → OBB（回転長方形）に改修
  - `checkWallCollision()`（移動）
  - `isBulletBlockedByWall()`（弾）
  - `isPointInBush()`（隠密判定）
  - スポーン・アイテム配置の検証
  - クライアント描画
- SAT（分離軸定理）ベースに移行
- コード変更は大きいが、パフォーマンスへの影響は軽微（壁数百個程度なら問題なし）

### ブリッジ × リバーの通行判定

Wall 型に `passable?: boolean` を追加:
```typescript
type Wall = {
  x: number; y: number;
  width: number; height: number;
  type?: WallType;
  rotation?: number;     // 自由角度（度）
  passable?: boolean;    // ブリッジ用: true なら通行許可ゾーン
};
```

判定順:
1. passable ゾーン（ブリッジ）にいる？ → 通行OK、リバー判定スキップ
2. リバーにいる？ → 通行NG

### 動的ブッシュの仕様

| 項目 | 仕様 |
|------|------|
| 配置 | マップオブジェクトとして配置、ゲーム開始時に `dynamicBushes[]` に変換 |
| 移動 | ロープで引っ張り → 自機位置に重なる |
| 回数 | 無制限 |
| 隠密 | 敵味方問わず、ブッシュ内にいれば隠密状態 |
| 貫通 | 弾・ボム・フラッグ・アイテム・ロープ、すべて貫通 |
| 用途 | 移動式の隠れ蓑。自分で引き寄せて隠密状態になれる |

`isPointInBush()` は `Wall[]`（静的）と `dynamicBushes[]`（動的）の両方を参照。

### マップサイズ

- 幅・高さともに自由指定（最小 500、最大 3500）
- 理論上 500x3500 のような細長マップも作成可能

### マップ要素（エディタで編集可能）

- `objects[]` — プレハブオブジェクト配置
- `walls[]` — 自由配置の壁（静的ブッシュ含む）
- `bushes[]` — 動的ブッシュ配置
- `spawnPoints[]` — チームごとのリスポーン地点
- `flagPositions[]` — CTF用の旗配置
- `itemSpawns` — アイテム初期配置（下記参照）

### アイテムスポーン

マップ作成者が選べる2モード:

**random モード（従来方式）**
- 位置も種類もランダムで初期配置

**manual モード（手動配置）**
- エディタでアイテムを1個ずつ配置 → そのまま初期局面になる
- 開幕の戦略的配置をデザイン可能

どちらのモードでも、取得後の再スポーンは従来通りランダム位置。

```json
{
  "itemMode": "manual",
  "itemSpawns": [
    { "x": 300, "y": 400, "type": "medic" },
    { "x": 600, "y": 200, "type": "bomb" }
  ]
}
```

### 制限（バランス＆パフォーマンス）

- 自由配置 Wall: 最大 **10個**
- プレハブオブジェクト: 最大 **100個**
- 展開後の Wall 総数: 上限 **500個**（重いプレハブ大量配置への安全弁）

プレハブ定義を保存（展開後ではなく）→ DB軽量:

{

### 実装順序

1. **プレハブ定義** — 20種の形状データを `shared/` に定義
2. **展開ロジック** — `objects[]` → `Wall[]` への展開関数
3. **OBB当たり判定** — AABB → OBB に `collision.ts` を改修（SAT導入）
4. **クライアント描画** — 回転対応 + プレハブtype別の描画
5. **マップ2枚作成** — プレハブを使って楽しいマップを組む（`shared/src/maps.ts` に直書き）
6. **（後で）DB保存 / マップエディタ** — 上記が動いてから着手
